# 新词本插件可优化项整理

本文档整理当前插件在功能、性能、权限、发布质量和成本方面的可优化点，并保留可行实现方式，便于后续迭代时选择。

## 1. 当前架构简述

本项目当前由以下部分组成：

- [background.js](../background.js)：登录、Firestore 读写、翻译请求、右键菜单
- [popup.js](../popup.js)：词库 UI、搜索、排序、翻译入口、设置页
- [popup.html](../popup.html) / [popup.css](../popup.css)：弹窗界面
- [content.js](../content.js) / [content.css](../content.css)：页面内提示逻辑
- [manifest.json](../manifest.json)：权限和 Chrome 扩展配置

词库数据结构当前为：

```text
users/{uid}/words/{wordId}
```

每个文档包含：单词、次数、收藏时间、上下文、翻译、近义词、反义词等。

## 2. 优先级最高的优化项

## 2.1 降低词库全量读取成本

### 当前情况

在 [popup.js](../popup.js) 初始化时，弹窗会调用 `GET_ALL_WORDS`；后台在 [background.js](../background.js) 中执行 `getAllWords(uid)`，一次性拉取最多 500 个文档。

这会带来两个问题：

- 用户词库越大，打开弹窗越慢
- Firestore 文档读取成本随“词库大小 × 打开频率”增长

### 优化方向 A：分页加载

实现思路：

- 首屏只加载最近 50 或 100 个单词
- 滚动到底部或点击“加载更多”时继续请求下一页

优点：

- 成本下降明显
- 首屏更快

缺点：

- 需要在 Firestore 层引入更稳定的排序字段和分页游标

### 优化方向 B：本地缓存 + 增量刷新

实现思路：

- 首次全量读取后，将结果缓存到 `chrome.storage.local`
- 后续打开弹窗先读取本地缓存，再按时间戳或版本做增量同步

优点：

- 用户感知更快
- 大幅减少重复读取

缺点：

- 本地与远端一致性管理更复杂

### 建议

先做分页，再考虑增量缓存。分页复杂度更低，可直接降低读取量。

## 2.2 收窄 Host Permissions

### 当前情况

[manifest.json](../manifest.json) 当前包含：

- `https://*.firebaseio.com/*`
- `https://firestore.googleapis.com/*`
- `https://identitytoolkit.googleapis.com/*`
- `https://securetoken.googleapis.com/*`
- `https://*.googleapis.com/*`
- `https://api.dictionaryapi.dev/*`
- `https://api.mymemory.translated.net/*`

其中 `https://*.googleapis.com/*` 范围偏宽。

### 可选优化方式

#### 方式 A：删除 `*.googleapis.com/*`

如果代码中实际上不需要额外访问该泛域，可以直接移除。

优点：

- 审核阻力更小
- 权限说明更清晰

风险：

- 如果后续引入更多 Google API，需要重新加权限并重新审核

#### 方式 B：拆分为更精确的域名

如果后续确实要调用 Google API，可改为具体 endpoint。

优点：

- 保留扩展能力
- 仍比泛域更容易说明

## 2.3 避免自动翻译造成的延迟与写入增长

### 当前情况

当前逻辑已经是“手动触发翻译”，这是合理的。但首次收藏时如果未来又改回自动翻译，会带来：

- 收藏操作变慢
- 外部 API 请求增多
- Firestore 写入变多

### 保留建议

继续保持当前策略：

- 收藏时只保存词和上下文
- 用户打开详情时再触发翻译

这是当前成本与体验之间比较平衡的方式。

## 3. 体验层优化项

## 3.1 登录失败与网络错误提示更细化

### 当前情况

[popup.js](../popup.js) 中登录失败时会展示通用错误信息，但对以下场景还可以更明确：

- OAuth 配置错误
- Firebase 授权域未配置
- 网络异常
- Firestore 读取失败

### 可选优化方式

#### 方式 A：细化错误映射

根据 `chrome.runtime.lastError` 和服务端错误码返回更可理解的中文提示。

#### 方式 B：增加“诊断提示”按钮

在登录失败面板中增加简单说明，例如：

- 请确认已联网
- 请重新打开扩展
- 请稍后再试

## 3.2 空状态与新用户引导

### 当前情况

已有空状态，但首次使用仍需要用户自己理解“去网页选词”。

### 可选优化方式

#### 方式 A：在空状态里加入 3 步引导

例如：

1. 打开英文网页
2. 选中英文单词
3. 点击收藏或右键收藏

#### 方式 B：加一个“打开示例页面”入口

可选一个固定英文页面供用户首次测试。

## 3.3 大词库下的搜索体验

### 当前情况

搜索逻辑主要在前端对全量数组处理。

### 可选优化方式

#### 方式 A：增加搜索防抖

优点：

- 输入更顺滑

#### 方式 B：优先搜索本地缓存

如果后续做本地缓存，搜索会更自然地获得收益。

## 4. 数据模型优化项

## 4.1 将上下文列表独立为子集合

### 当前情况

当前 `contexts` 直接存在单词文档里，最多保留最近 5 条。

优点：

- 简单
- 单词详情读取一次即可拿到上下文

缺点：

- 文档会越来越重
- 如果未来保留更多上下文，写入成本和文档大小都上升

### 可选方式

#### 方式 A：继续保留内嵌数组

适用于：

- 上下文条数少
- 不需要复杂分析

#### 方式 B：改成子集合

结构示例：

```text
users/{uid}/words/{wordId}/contexts/{contextId}
```

优点：

- 可无限扩展
- 支持分页查看上下文

缺点：

- 读取详情时要多一次查询

### 建议

当前阶段没必要拆。除非你后续明确要保留更多历史句子。

## 4.2 翻译缓存策略更明确

### 当前情况

翻译结果写回 Firestore，这本身已经是一种缓存。

### 可继续优化的方式

#### 方式 A：增加 `translationUpdatedAt`

优点：

- 更容易做过期策略
- 方便以后重刷翻译

#### 方式 B：记录翻译来源

例如：

- `translationSource: "mymemory"`
- `translationSource: "google-user-key"`

优点：

- 更利于排查翻译差异

## 5. 发布与审核优化项

## 5.1 权限最小化

### 当前情况

已经从“常驻 `<all_urls>` 注入”收窄到“按需动态注入页面提示”，这是正确方向。

### 后续可继续优化

- 重新梳理是否仍需要所有现有 host permissions
- 发布前为每个权限保持一份固定说明模板

## 5.2 商店元数据统一维护

### 当前问题

商店说明、隐私政策、测试说明容易与代码脱节。

### 可选方式

#### 方式 A：在仓库保留固定版本文案

例如在 `docs/` 下统一维护：

- 商店说明
- 测试说明
- 权限说明

#### 方式 B：每次发布时做 checklist

建议后续发布时检查：

- 是否新增权限
- 是否新增数据收集项
- 是否新增第三方服务
- 是否要同步更新隐私政策

## 6. 成本优化项

## 6.1 最主要的成本热点

当前最可能产生持续成本的不是翻译，而是 Firestore 读取。

原因：

- 每次打开弹窗都会拉全量词库
- 用户词库越大，读次数越高

### 低成本优化顺序建议

1. 分页读取
2. 本地缓存
3. 只在必要时刷新

## 6.2 减少不必要写入

当前已做得较好：

- 已有词只用 `updateMask` 局部更新
- 翻译不是每次收藏都自动触发

后续建议：

- 不要因为打开详情页就重复写翻译结果
- 如果翻译结果未变化，不要重复 PATCH

## 7. 推荐的后续迭代顺序

如果后续要继续优化，建议顺序如下：

1. 词库读取分页
2. 权限进一步精简
3. 错误提示与引导优化
4. 翻译来源与更新时间字段
5. 本地缓存与增量同步

这个顺序的好处是：

- 先降成本
- 再提审核通过率
- 最后再做复杂度更高的体验优化
