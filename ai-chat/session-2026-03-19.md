### 一、近义词 / 反义词

**数据来源**：Free Dictionary API（`api.dictionaryapi.dev`），已在 `manifest.json` 中授权，无需额外配置。

**实现逻辑**：

1. `background.js` 的 `fetchWordTranslations()` 在解析词典返回结果时，同时收集 `meaning` 级别和 `definition` 级别的 `synonyms` / `antonyms`，去重后各取最多 10 条
2. 翻译写入 Firestore 时新增 `synonyms`、`antonyms` 两个字段（`TRANSLATE_WORD` 消息的 `updateMask` 同步更新）
3. `popup.js` 新增 `renderRelatedWords(wordData)` 函数，在详情弹层翻译区域下方渲染近/反义词标签组

**Firestore 数据结构新增字段**：

```
users/{uid}/words/{wordId}
  + synonyms: array<string>   // 近义词，最多 10 条
  + antonyms: array<string>   // 反义词，最多 10 条
```

**修改文件**：

### 二、CEFR 词汇难度分级（已废弃）

**需求背景**：对收藏的单词标注 A1–C2 难度等级，帮助用户了解词汇水平。

**调研结论**：目前业界**没有免费的公开 CEFR 查询 API**。Cambridge EVP 和 Oxford Learner's Dictionary 均有 CEFR 标注，但均为商业授权，不对外提供接口。

**实现过程**：

1. 初版使用本地 `cefr-data.js`（约 300 词的静态词表）实现展示
2. 列表项在单词名后展示带颜色的等级徽章（A1 绿 → C2 红）
3. 详情页 meta 区域同步展示等级
4. 额外利用 Levenshtein 编辑距离，从词表中查找形近词展示在详情页

**废弃原因**：本地词表覆盖率过低（约 300 词），大多数生词无法匹配到等级，实用性差；且无可靠 API 替代方案。

**最终处理**：

- 删除 `cefr-data.js` 文件
- 从 `popup.js` 移除 `CEFR_DATA` import、`CEFR_COLORS`、`getCefrLevel`、`getCefrBadgeHtml`、`levenshtein`、`getSimilarWords` 所有函数
- 从 `renderRelatedWords()` 移除形近词逻辑和各标签上的 CEFR 角标
- 从 `popup.html` 移除 `<span id="detail-cefr">` 节点
- 从 `popup.css` 移除 `.cefr-badge`、`.cefr-badge-compact`、`.word-text-row`、`.related-tag-sim`、`.related-cefr` 样式

### 四、词库分页加载 + 搜索防抖 + 提示文案优化

**需求**：

- 词库分页加载，每页 50 条
- 默认按收藏次数排序
- 关键词搜索增加防抖
- 登录失败、网络异常、列表读取失败等提示更人性化

**实现思路**：

1. 弹窗首屏只请求一页数据，默认 `PAGE_SIZE = 50`
2. 前端保留当前页缓存与全量缓存两套数据：

- 普通浏览词库时只加载当前页
- 搜索和导出时按当前排序补齐全部分页数据

3. 搜索输入采用 `250ms` 防抖，避免每次按键都立即触发过滤或远端补数
4. 错误提示统一走 `getFriendlyErrorMessage()`，映射登录取消、网络问题、OAuth/Firebase 配置不完整、权限不足、登录态过期、后台无响应等场景

**UI 改动**：

- 词库列表底部新增分页栏：上一页 / 下一页 / 当前页信息
- 新增加载失败态，展示「重新加载」按钮
- 页眉统计文案改成分页上下文，例如：
  - 普通浏览：`第 N 页 · 50 条`
  - 搜索结果：`匹配 X 条 · 第 N / M 页`

**前端状态设计**：

```js
PAGE_SIZE = 50;
SEARCH_DEBOUNCE_MS = 250;

pageCache; // 当前排序下的页缓存
pageTokenCache; // 下一页 Firestore pageToken 缓存
fullWordCache; // 搜索 / 导出时使用的全量缓存
```

**修改文件**：

| 文件            | 改动说明                                                                                                    |
| --------------- | ----------------------------------------------------------------------------------------------------------- |
| `background.js` | 新增 `GET_WORDS_PAGE`；词库读取支持分页、排序；保留 `getAllWords()` 供清空/导出等全量场景调用               |
| `popup.js`      | 列表加载改为分页；新增 `pageCache` / `pageTokenCache` / `fullWordCache`；搜索改为防抖；新增统一错误文案映射 |
| `popup.html`    | 新增 `list-error-state` 加载失败态；新增 `pagination-bar` 分页栏                                            |
| `popup.css`     | 新增分页栏、重试按钮、次级按钮样式                                                                          |

### 五、分页接口排障与修复

**问题现象**：

- 功能改完后，弹窗持续显示「获取生词列表失败」
- 首屏无法拿到词库数据

**初始实现的问题点**：

最初分页实现使用了 Firestore REST 的 `runQuery` 风格请求：

```text
.../documents/users/{uid}:runQuery
```

并在 `structuredQuery` 里使用：

- `from: [{ collectionId: "words" }]`
- `orderBy`
- `offset`
- `limit`

这套写法在当前项目这条用户子集合路径下请求不稳定，导致词库列表一直读失败。

**最终修复方案**：

改为 Firestore 文档列表接口：

```text
.../documents/users/{uid}/words?pageSize=50&orderBy=count desc&pageToken=...
```

分页方式也同步从“页码 + offset”改为“`nextPageToken` 游标翻页”：

1. 后台 `queryWordsPage()` 返回：

- `words`
- `nextPageToken`

2. `GET_WORDS_PAGE` 消息返回：

- `words`
- `hasNextPage`
- `nextPageToken`

3. 前端在 `pageTokenCache` 中缓存各页对应的下一页游标，翻页时带上对应 token 请求

**修复后的收益**：

- 词库分页恢复正常
- 不再依赖 `offset` 做深分页
- 搜索和导出补齐全量数据时，也统一复用同一套 `pageToken` 逻辑

## 当前 Firestore 数据结构（完整）

```
users/{uid}/words/{word小写}
  - word: string                    // 单词（小写）
  - displayWord: string             // 原始大小写
  - count: number                   // 收藏次数
  - firstCollectedAt: timestamp
  - lastCollectedAt: timestamp
  - contexts: array[{               // 最近 5 条上下文
      text, url, title, time
    }]
  - contextTranslation: string      // 上下文中文翻译
  - translations: array[{           // 词性释义
      pos, posZh,
      definitions: [{ en, zh }]
    }]
  - synonyms: array<string>         // 近义词（最多 10 条）
  - antonyms: array<string>         // 反义词（最多 10 条）
```
